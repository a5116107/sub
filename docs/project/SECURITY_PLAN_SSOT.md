# 安全方案 SSOT（防逆向 / 防泄露 / 防 DDoS）

最后更新：2026-02-01  
适用范围：Sub2API（前端 + 后端 + 运维部署）  
目标读者：项目负责人 / 后端 / 运维（管理员不外包运营，允许管理端查看明文 secrets）

---

## 0. 背景与结论（先说人话）

你们选择的 **Option C（弱保密）= 自建完整网关** 的核心价值是：

- **对外只暴露一个稳定 API 面**（Sub2API），把所有上游账号、路由策略、额度/计费规则都收敛在服务端。
- **能显著减少“意外泄露”**（前端/SDK/错误消息/日志/接口返回把上游信息带出去）。
- 但它 **无法阻止“有权限的人最终拿到 secrets”**（例如管理员、主机被攻陷、运行环境被读取）。所以必须靠权限隔离、审计、密钥生命周期与运维边界来降低风险。

这份文档的重点不是“对抗国家级逆向”，而是：**让一个普通下游用户（拿到 API Key、抓包、读前端代码、逆向客户端）无法获得上游的任何可复用信息**（上游域名、账号标识、token、路由链路、策略细节等）。

---

## 1. 威胁模型（Threat Model）

### 1.1 资产（Assets）

- 上游 secrets：access_token / refresh_token / session key / client_secret / cookie 等
- 上游链路与策略：上游域名、账号池结构、路由/调度策略、proxy 策略、失败降级策略
- 运营资产：价格/倍率、配额、风控策略、用户分组策略
- 数据资产：用户隐私、日志、usage 统计、账务数据

### 1.2 对手（Adversaries）

- A1：未登录互联网攻击者（扫接口、撞库、爬虫、DDoS）
- A2：普通付费/免费用户（拿到自己的 API Key、抓包、看前端 bundle、尝试调用内部 API）
- A3：被盗号用户（拿到用户账号/Token）
- A4：内部人员/管理员（可看明文 secrets，属于“高权限可见”）
- A5：主机/容器被入侵（读进程内存、读 /app/data、读数据库）

### 1.3 非目标（Non-goals）

- 不做“客户端不可逆”的幻想：**前端/APP/JS 最终都可被逆向**。防逆向的正确目标是：即使被逆向，也拿不到上游 secrets/链路。
- 不在应用层单独对抗大规模 DDoS：需要 CDN/WAF/边缘限流配合。

## 2. 现状盘点（已存在的安全机制）

以下为代码层面可见的机制（并非完整清单）：

- **权限隔离**：管理端功能走 `/api/v1/admin/*`（需管理员鉴权）；普通用户只见自己的 Key/用量等。
- **日志脱敏**：`backend/internal/util/logredact` 对常见敏感字段（token/secret/password 等）做 JSON 级脱敏。
- **认证风控**：认证相关接口已接入 Redis 速率限制（例如 forgot/reset/promo validate 等）`backend/internal/server/routes/auth.go`。
- **安全头**：服务端统一下发安全相关 headers（CSP/等）。
- **URL Allowlist**：支持对上游/定价等 URL 做 allowlist 校验（避免 SSRF/错误配置连到不该连的内网地址）。
- **索引控制**：新增 `robots.txt/sitemap.xml` 与 `X-Robots-Tag` 策略，默认不被搜索引擎索引（需显式开启 public 模式）。

## 3. 防逆向 / 防泄露：核心原则（必须严格执行）

### 3.1 “上游信息”严禁出现在任何下游可见面

下游可见面包括但不限于：

- 前端静态资源（JS/CSS/SourceMap）、前端配置注入、页面源码
- 任意公开 API 返回（包括错误信息、调试字段）
- 网关 `/v1` 错误响应（429/5xx 等）
- 日志与监控导出（如果用户可见）

### 3.2 错误与日志分层

- 对下游：只返回 **抽象错误**（例如 `upstream_error`、`rate_limited`、`temporarily_unavailable`）
- 对内部：记录 **详细错误**，但必须做脱敏与结构化（避免把 token、cookie、完整 URL 打进日志）

### 3.3 Secrets 只存在服务端 + 最小暴露面

- 普通用户永远不应拿到上游 secrets
- 管理员允许查看明文，但要：
  - 默认不展示，必须点击“显示/复制”才展示
  - 记录审计（谁在什么时间查看/复制了哪类 secret）
  - 建议强制开启 2FA（管理端）

## 4. 设计与实施：Option C（弱保密）落地路线图

### Phase 0（立刻落地，1–3 天）：阻断“意外泄露”

1) **API 输出审计**
- 审计所有对外 DTO/JSON：确保无上游域名、账号标识、proxy 链路字段外泄
- 对错误响应做统一收敛：对外错误禁止透出 upstream message/URL

2) **前端资源审计**
- 生产环境禁用 sourcemap（或仅内网可访问）
- 前端配置注入只允许公开字段（如 site_name/doc_url），严禁注入任何上游信息

3) **日志与监控**
- 扩展脱敏键集合：覆盖 `authorization`, `api_key`, `x-api-key`, `cookie` 等（按实际日志格式补齐）
- 统一日志规范：禁止打印完整请求体（尤其是上游鉴权相关）

4) **管理端明文策略（符合你们偏好）**
- 允许明文展示，但必须：显式点击显示 + copy 时二次确认（可选）+ 审计记录（P1 可做）

验收标准：
- 普通用户在浏览器 DevTools / 抓包 / 逆向 bundle 后，找不到任何上游域名/账号/secret/链路字段。

### Phase 1（短期，1–2 周）：把“逆向”变成无意义

1) **上游凭证加密存储（推荐）**
- 数据库中存密文；服务端用主密钥（ENV/KMS）解密
- 管理端仍可查看明文（通过解密 + 权限判断）

2) **上游链路最小化**
- 对下游只暴露“能力/模型”层面的抽象名称，不暴露上游供应商/账号池结构（除非产品要求）

3) **审计与告警**
- 管理端查看/复制 secrets 事件审计
- 异常行为告警：登录失败、爆破、异常高频调用、异常退款/兑换等

验收标准：
- DB 泄露情况下（无主密钥），攻击者无法直接复用上游凭证。

### Phase 2（中期，>2 周）：对抗更强对手（可选）

- 细粒度 RBAC：把“可查看明文 secret”做成独立权限
- 密钥轮换：定期轮换上游 token/密钥；最短可用窗口
- 多层风控：设备指纹、地理/ASN 异常、策略引擎

## 5. 防 DDoS / 防滥用：闭环方案

### 5.1 分层防护（推荐组合）

1) **边缘层（首选）**
- Cloudflare / Nginx / CDN：基于 IP/ASN/UA 的限流与挑战（JS challenge / Turnstile）
- 保护路径：`/api/*`, `/v1/*`, `/login`, `/register`, `/docs/*`

2) **应用层（Sub2API）**
- 认证相关接口已有限流：继续补齐 login/register/send-code 的限流策略
- `/v1` 网关层：按 **API Key + IP** 做双维度限流（Redis token bucket），并设置 fail-close 策略
- 并发控制：对单 Key/单用户/单 IP 设置并发上限

3) **数据层**
- Redis/DB 连接池限额与超时（避免被慢请求拖死）

### 5.2 验收标准（建议压测）

- 模拟 3 类压测：未登录打 `/login`、有 Key 打 `/v1`、爬虫扫 `/docs`
- 目标：系统保持可用，关键业务接口 P95 可控，数据库无雪崩

---

## 6. 运维边界（非常关键）

- 管理端不外包：意味着“管理员侧防泄露”主要是**防误操作与审计**，而不是防内部对手。
- 建议至少做到：
  - 管理员强制 2FA
  - 数据库与 /app/data 做备份加密
  - 主机最小权限、容器只读文件系统（可选）、禁止把 secrets 写入镜像与日志
