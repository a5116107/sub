## SEC-038 — Security: remove hardcoded OAuth client secrets (Gemini CLI / Antigravity)

### Problem
The repo contains hardcoded Google OAuth `client_secret` values. This is credential leakage and creates a supply-chain risk (anyone with code access can reuse the secret). It also makes secret rotation difficult and encourages insecure defaults.

### Evidence (code)
- Gemini CLI built-in OAuth secret constant: `backend/internal/pkg/geminicli/constants.go:42`
- Gemini OAuth service compares against the built-in secret constant: `backend/internal/service/gemini_oauth_service.go:84`, `backend/internal/service/gemini_oauth_service.go:158`
- Antigravity OAuth secret constant: `backend/internal/pkg/antigravity/oauth.go:23`
- Antigravity OAuth token exchange sends `client_secret` param from constant: `backend/internal/pkg/antigravity/client.go:191`, `backend/internal/pkg/antigravity/client.go:230`

### Goal
Remove committed OAuth client secrets and require operators to provide any secrets via runtime configuration (env/config), while keeping behavior explicit and testable.

### Acceptance Criteria
- [ ] Remove hardcoded OAuth client secrets from code (no `GOCSPX-` style values committed).
- [ ] Gemini CLI “built-in client” secret must be provided at runtime (env/config) if that fallback is used.
- [ ] Antigravity OAuth client secret must be provided at runtime (env/config) for token exchange/refresh.
- [ ] Error messages for missing secrets are explicit and do not log the secret.
- [ ] Verification: `cd backend; go test ./...` PASS.

### Implementation Notes (done)
- Gemini CLI built-in secret is now read from `GEMINICLI_BUILTIN_OAUTH_CLIENT_SECRET` (and optional `GEMINICLI_BUILTIN_OAUTH_CLIENT_ID`).
- Antigravity OAuth secret is now read from `ANTIGRAVITY_OAUTH_CLIENT_SECRET` (and optional `ANTIGRAVITY_OAUTH_CLIENT_ID`).
