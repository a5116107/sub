## SEC-040 — Security/SSRF: make URL allowlist enforcement default-on (secure-by-default)

### Problem
The current default config disables URL allowlist/SSRF checks (`security.url_allowlist.enabled=false`). In this mode, multiple gateway services fall back to `ValidateURLFormat` which only performs minimal URL parsing + scheme validation, and does not enforce allowlist or DNS-resolved IP safety. This enables SSRF-style abuse via user-controlled `base_url` (stored in account credentials) if an attacker can create/modify accounts (or if admin compromise occurs).

Even though the code logs a warning when allowlist is disabled, the behavior is still insecure-by-default.

### Evidence (code)
- Default disables allowlist: `backend/internal/config/config.go:899`
- When allowlist disabled, base_url validation becomes minimal:
  - `backend/internal/service/openai_gateway_service.go:1916`
  - `backend/internal/service/gateway_service.go:4366`
- User-controlled base_url surfaces:
  - `backend/internal/service/iflow_gateway_service.go:98`
  - `backend/internal/service/qwen_gateway_service.go:281`
  - `backend/internal/service/openai_gateway_service.go:1305`
  - `backend/internal/service/account_test_service.go` (uses account base_url for test calls)

### Goal
Make SSRF defenses the default by enabling `security.url_allowlist.enabled` by default. Operators can still explicitly disable it for debugging, but production defaults must be safe.

### Acceptance Criteria
- [ ] Change viper default `security.url_allowlist.enabled` to `true`.
- [ ] Ensure existing tests still pass (or update tests to reflect new secure default).
