## SEC-045 — Security/SSRF: split `upstream_hosts` allowlist into provider-specific lists

### Problem
The current allowlist uses a single `security.url_allowlist.upstream_hosts` list shared across multiple upstream validators (OpenAI/Anthropic/Gemini/iFlow/Qwen). This creates an overly broad trust boundary: if an operator adds a host needed for one provider, they implicitly allow that host to be used for other providers' `base_url`, increasing SSRF blast radius and the chance of accidental misconfiguration.

### Evidence (code)
The following services all use `cfg.Security.URLAllowlist.UpstreamHosts`:
- `backend/internal/service/openai_gateway_service.go:1924`
- `backend/internal/service/gateway_service.go:4374` (Anthropic/Claude paths)
- `backend/internal/service/gemini_messages_compat_service.go:404`
- `backend/internal/service/iflow_gateway_service.go:60`
- `backend/internal/service/qwen_gateway_service.go:76`
- Account base_url validation uses the same list when allowlist enabled: `backend/internal/service/account_service.go:363`

### Goal
Split upstream allowlist into provider-specific lists so each platform only accepts hosts explicitly intended for it.

### Proposed Config
Under `security.url_allowlist`:
- `openai_hosts`
- `anthropic_hosts`
- `gemini_hosts`
- `iflow_hosts`
- `qwen_hosts`

### Acceptance Criteria
- [ ] Add provider-specific allowlist fields to config with safe defaults.
- [ ] Update each provider validator to use its specific allowlist list (not the shared upstream_hosts).
- [ ] Keep backward compatibility: if a provider-specific list is empty, fall back to `upstream_hosts` (so existing configs don’t break immediately).
- [ ] Update tests to cover the new selection logic.
- [ ] Verification: `cd backend; go test ./...` PASS.

### Implementation Notes (done)

  - `openai_hosts`, `anthropic_hosts`, `gemini_hosts`, `iflow_hosts`, `qwen_hosts`
- Validators now prefer the provider-specific list and fall back to `upstream_hosts` when empty.
