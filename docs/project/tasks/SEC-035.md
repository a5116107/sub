## SEC-035 — Security/DoS: cap remaining external HTTP response bodies (Claude usage / Gemini RM / Antigravity)

### Problem
There are still a few external HTTP call paths that use `io.ReadAll(resp.Body)` without a size cap. A large response body (even on error paths) can cause high memory usage and DoS.

### Evidence (code)
- Claude OAuth usage fetcher reads unbounded error body: `backend/internal/repository/claude_usage_service.go:97`
- Gemini OAuth: GCP Resource Manager projects list reads unbounded: `backend/internal/service/gemini_oauth_service.go:1036`
- Antigravity client reads unbounded in several endpoints:
  - `backend/internal/pkg/antigravity/client.go:210`
  - `backend/internal/pkg/antigravity/client.go:247`
  - `backend/internal/pkg/antigravity/client.go:278`
  - `backend/internal/pkg/antigravity/client.go:331`
  - `backend/internal/pkg/antigravity/client.go:419`

### Goal
Add bounded reads for these remaining call sites with reasonable max sizes and deterministic oversize errors.

### Acceptance Criteria
- [ ] Claude usage fetch caps response body (both non-200 and 200 JSON parse).
- [ ] Gemini resource manager projects list caps response body.
- [ ] Antigravity client caps OAuth/token/userinfo + v1internal JSON response bodies.
- [ ] Tests cover “response too large” for each area.
- [ ] Verification: `cd backend; go test ./...` PASS.
