# SEC-030: XSS: harden public HomeContent rendering (v-html)

> Status: ✅ Verified (2026-02-02)
> 验证：`cd backend; go test ./...` PASS；`pnpm -C frontend test:run` PASS；`pnpm -C frontend build` PASS

## 背景/问题（以代码为准）

首页支持通过公开 settings 下发 `HomeContent` 并在前端使用 `v-html` 渲染：

- 证据：
  - `backend/internal/handler/setting_handler.go`：`GetPublicSettings` 返回 `HomeContent`
  - `frontend/src/views/HomeView.vue`：`<div v-else v-html="sanitizedHomeContent"></div>`（DOMPurify）

风险：
- 一旦后台账号/设置被篡改（或管理面存在 XSS/CSRF/弱口令等），此处会变成稳定的存储型 XSS 面（公开可访问）。
- 当前代码注释“接受风险”，但安全层面仍建议提供可选的安全阈值/默认防护。

## 目标

- 降低 XSS/内容注入风险，至少提供默认安全策略（或可配置开关），不影响正常公告展示。

## 方案选项（A/B/C）

| 选项 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| A (最小改动) | 后端对 `HomeContent` 做 HTML sanitize（允许安全标签/属性）再下发 | 默认更安全；前端无感 | 需引入 sanitize 依赖/规则需维护 |
| B | 仅允许 iframe URL（HTML 直接拒绝/转义） | 风险最低 | 会破坏现有 HTML 公告能力 |
| C | 前端侧 sanitize 后再 v-html（DOMPurify 等） | 前端可控；规则容易迭代 | 仍需依赖；需保证 SSR/构建一致性 |

推荐：优先 A（后端统一兜底），并提供显式开关允许“原样 HTML”（强提醒风险）。

## 已实现（选项 C）

- 前端对 `homeContent` 先做 DOMPurify sanitize，再通过 `v-html` 渲染，避免脚本/事件属性执行。

## 验收标准

- [ ] `HomeContent` 默认不会执行脚本/事件属性（如 `onerror`/`onclick`）
- [ ] 公告常用样式（链接/列表/粗体/换行）可正常展示
- [ ] 如新增开关：默认安全，只有管理员显式开启才允许 raw HTML
