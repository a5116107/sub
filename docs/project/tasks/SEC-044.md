## SEC-044 — Security/SSRF: enforce URL allowlist for iFlow/Qwen base_url (and port/path restrictions)

### Problem
`iFlow` and `Qwen` gateway services validate `base_url` with `RequireAllowlist: false`, so any public HTTPS host passes. If an attacker can influence account `base_url` (admin compromise, unsafe admin workflow, etc.), this becomes a generic SSRF forwarding primitive (even with other defaults tightened).

Additionally, these validators currently do not enforce the stricter `AllowPorts: [443]` and `RequireNoPath: true` policy used by OpenAI/Gateway validators (SEC-043).

### Evidence (code)
- iFlow `RequireAllowlist: false`: `backend/internal/service/iflow_gateway_service.go:49`
- Qwen `RequireAllowlist: false`: `backend/internal/service/qwen_gateway_service.go:65`
- Account credential validation for these platforms also uses non-allowlisted `validateOptionalPublicHTTPSBaseURL`: `backend/internal/service/account_service.go:429`, `backend/internal/service/account_service.go:456`

### Goal
Bring iFlow/Qwen base_url validation in line with the unified URL allowlist policy:
- Require allowlist when `security.url_allowlist.enabled=true`
- Enforce `AllowPorts: [443]` and `RequireNoPath: true`
- Keep behavior predictable when base_url is empty (use service defaults)

### Acceptance Criteria
- [ ] iFlow validateUpstreamBaseURL uses `AllowedHosts: cfg.Security.URLAllowlist.UpstreamHosts`, `RequireAllowlist: true`, `AllowPrivate: cfg.Security.URLAllowlist.AllowPrivateHosts`, `AllowPorts: [443]`, `RequireNoPath: true`.
- [ ] Qwen validateUpstreamBaseURL uses same allowlist policy.
- [ ] AccountService.TestCredentials for iFlow/Qwen validates base_url using allowlist-based validation (not just public HTTPS).
- [ ] Tests cover rejecting non-allowlisted base_url and rejecting port/path.
