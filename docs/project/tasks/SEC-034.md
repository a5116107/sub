## SEC-034 — Security/DoS: cap upstream non-streaming response bodies in gateway services

### Problem
Several gateway success paths read upstream HTTP response bodies with `io.ReadAll(resp.Body)` without a size cap. If an upstream endpoint (or misconfiguration) returns an arbitrarily large body, the server can be forced into high memory usage (DoS).

### Evidence (code)
- Claude gateway non-streaming: `backend/internal/service/gateway_service.go:3665`
- Claude `count_tokens` forward: `backend/internal/service/gateway_service.go:4159` (+ retry reads)
- OpenAI gateway non-streaming: `backend/internal/service/openai_gateway_service.go:1758`
- Gemini v1internal forward non-streaming: `backend/internal/service/gemini_v1internal_forward.go:295`
- iFlow non-streaming success path: `backend/internal/service/iflow_gateway_service.go:262`
- Qwen non-streaming success path: `backend/internal/service/qwen_gateway_service.go:442`
- Gemini native non-streaming: `backend/internal/service/gemini_messages_compat_service.go:2255`

### Goal
Introduce a shared bounded read helper for upstream non-streaming bodies and apply it in the above call sites.

### Acceptance Criteria
- [ ] All listed call sites use a bounded read (LimitReader + oversize check).
- [ ] Oversized upstream responses return a deterministic gateway error (no partial forwarding).
- [ ] Unit tests cover the bounded read helper’s oversize behavior.
- [ ] Verification: `cd backend; go test ./...` PASS.
