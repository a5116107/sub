## SEC-037 — Security/DoS: apply RequestBodyLimit to /api/v1 and return 413 for webhooks

### Problem
`/api/v1` routes currently do not apply a request body size limit middleware. Some endpoints (e.g. payment webhooks) read the entire request body (`io.ReadAll`) and can be abused with oversized requests for memory DoS.

### Evidence (code)
- `/api/v1` group is created without `RequestBodyLimit`: `backend/internal/server/router.go:70`
- PayPal/Creem webhook handlers read full body: `backend/internal/handler/payment_handler.go:126`, `backend/internal/handler/payment_handler.go:140`

### Goal
Add a configurable body limit for `/api/v1` and ensure oversized webhook bodies return HTTP 413 with a clear message.

### Acceptance Criteria
- [ ] New config `server.api_max_body_size` with safe default (>= model pricing import needs).
- [ ] `/api/v1` router group uses `RequestBodyLimit(server.api_max_body_size)`.
- [ ] PayPal/Creem webhooks return `413 Request Entity Too Large` when MaxBytesError is hit.
- [ ] Tests cover oversized webhook behavior.
- [ ] Verification: `cd backend; go test ./...` PASS.
