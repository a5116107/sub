## SEC-043 — Security/SSRF hardening: restrict base_url ports and paths (reduce allowlist bypass)

### Problem
Even with host allowlisting enabled, `base_url` validation currently only checks scheme/host (and optionally private host) and does not enforce safe defaults for:
- Non-default ports (e.g., `:80`, `:8080`, `:8443`) which can target different services on the same host.
- Unexpected paths (e.g., allowing `https://api.openai.com@evil.com` is already blocked by URL parsing, but allowing arbitrary path prefixes can still widen attack surface on allowlisted domains).

Given this gateway forwards user-controlled request payloads, minimizing `base_url` flexibility reduces SSRF blast radius and prevents allowlist “same host different service” issues.

### Evidence (code)
- Host allowlist checks only hostname; port is accepted if present: `backend/internal/util/urlvalidator/validator.go:42`
- base_url validated and then concatenated with fixed endpoint paths:
  - iFlow: `backend/internal/service/iflow_gateway_service.go:103`
  - Qwen: `backend/internal/service/qwen_gateway_service.go:286`
  - OpenAI/Gateway: validateUpstreamBaseURL then append endpoint paths.

### Goal
Enforce a conservative base_url policy by default:
- Disallow explicit ports unless allowlisted by config (or only allow standard 443 for https).
- Normalize/forbid base_url paths where the service expects a fixed path prefix.
